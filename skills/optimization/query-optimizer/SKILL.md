# Query Optimizer

Use this skill when database interactions are slow, causing timeouts, high CPU load, or application sluggishness.

## Triggers

- User reports slow API endpoints or dashboard loading times.
- Logs show queries taking hundreds of milliseconds or seconds.
- Reviewing ORM code (Prisma, Django, TypeORM, SQLAlchemy).
- Fixing N+1 query problems in GraphQL or REST endpoints.
- Task asks to "optimize DB", "add indexes", or "reduce database load".

## Inputs To Inspect

- Database schema definitions (SQL, Prisma schema, Django models).
- Code executing the queries (repositories, controllers, resolvers).
- Actual SQL generated by the ORM (enable query logging).
- Loop structures making database calls inside them.

## Optimization Strategy

1. **Find N+1 Queries**: Look for loops executing queries or ORM lazy-loading.
2. **Analyze Execution**: Use `EXPLAIN ANALYZE` to find sequential scans.
3. **Add Indexes**: Target `WHERE`, `JOIN`, and `ORDER BY` columns.
4. **Limit Data**: Enforce pagination and select only required columns.
5. **Batch/Cache**: Batch updates or cache expensive analytical queries.

## 1. Resolving N+1 Queries

An N+1 query occurs when you load a list of items, then run a separate query for each item's relations.

**❌ Bad (N+1):**
```typescript
// Fetches 10 users (1 query)
const users = await prisma.user.findMany({ take: 10 });

// Fetches posts for each user (10 additional queries!)
for (const user of users) {
  user.posts = await prisma.post.findMany({ where: { userId: user.id } });
}
```

**✅ Good (JOIN / Eager Loading):**
```typescript
// Fetches users AND their posts in a single query
const users = await prisma.user.findMany({
  take: 10,
  include: { posts: true }
});
```

*For GraphQL:* Implement dataloaders to batch nested relation queries.

## 2. Using EXPLAIN

Do not guess which index to add. Ask the database.

```sql
-- PostgreSQL
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 123 AND status = 'pending';
```
Look for `Seq Scan` (Sequential Scan) on large tables. This means the DB is reading every row.

## 3. Targeted Indexing

Add indexes to columns used in lookups or sorting.

```sql
-- Good: Index exactly what is queried
CREATE INDEX CONCURRENTLY idx_orders_user_status ON orders (user_id, status);

-- Good: Index for ordering
CREATE INDEX CONCURRENTLY idx_orders_created_desc ON orders (created_at DESC);
```

*Note: Always use `CONCURRENTLY` in Postgres to avoid locking the table during index creation.*

## 4. Column Selection & Pagination

Stop requesting `SELECT *` if you only need the ID and name.

```typescript
// Prisma: select specific fields
const users = await prisma.user.findMany({
  select: { id: true, name: true }
});
```

Enforce limits to protect against unbounded queries:
```typescript
// Always require a limit
const take = Math.min(req.query.limit || 20, 100); 
```

## Common Pitfalls

- Over-indexing: adding an index to every column slows down `INSERT` and `UPDATE` operations significantly.
- Using `WHERE LOWER(email) = '...'` negates standard indexes. Use a functional index (`CREATE INDEX ON users (LOWER(email))`) or Postgres `citext`.
- Testing optimizations on an empty local database. Always test or analyze against production-scale data volumes.
- Missing `LIMIT` on background job queries, loading millions of rows into RAM.

## Done Criteria

- Loops executing queries are replaced with IN clauses or JOINs.
- Slow queries are supported by appropriate composite or single-column indexes.
- ORM code generates predictable, bounded SQL.
- Unnecessary columns are omitted from the SELECT clause.
